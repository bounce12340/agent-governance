# Example: App Launch Law / 範例：APP 上架法律

## English

This example shows how a request becomes a governed app-launch case.

### Constitution check

- Privacy is mandatory.
- No red line may be bypassed.
- Harness evidence is required before judgment.

### Legislative output

- Target: build a consumer app in 4 to 8 weeks.
- Market research: at least 30 valid responses and 5 interviews.
- MVP: one core flow, one account system, one persistent storage path.
- Red lines: privacy policy, no deceptive UI, no forbidden payment bypass.
- Rework cap: no more than 3 failed review cycles.

### Executive output

- Implement the core flow.
- Build a working iOS/Android-ready app.
- Provide logs, screenshots, and a test checklist.
- Prepare harness artifacts before review.

### Harness gate / Evidence gate

- Test plan exists.
- Evidence bundle exists.
- Failure mode notes exist.
- If any are missing, the case is incomplete.

### Judiciary check

- Does the product match the law?
- Does the core flow pass the test harness?
- Are the red lines respected?
- Is the constitution satisfied?

### Pass criteria

- All required fields present.
- Core flow succeeds without crash.
- No policy violations.
- Harness gate passed.

## 繁體中文

這個範例示範如何把需求變成一個有治理流程的 APP 上架案件。

### 憲法檢查

- 隱私必須存在。
- 紅線不能被繞過。
- 在司法判定前，必須先有 Harness 證據。

### 立法權輸出

- 目標：在 4 到 8 週內做出一個可上架的消費型 APP。
- 市場調查：至少 30 份有效回覆、5 場訪談。
- MVP：一條核心流程、一套帳號系統、一個資料持久化路徑。
- 紅線：隱私政策、不能誤導 UI、不能繞過支付規範。
- Rework 上限：最多 3 次失敗修正。

### 行政權輸出

- 實作核心流程。
- 做出可供 iOS / Android 上架的版本。
- 提供 log、截圖、測試清單。
- 在審查前先準備好 harness 證據包。

### Harness gate / 證據門檻

- 需要有 test plan。
- 需要有 evidence bundle。
- 需要有 failure mode notes。
- 任何一項缺少，都視為案件未完成。

### 司法權審查

- 產品是否符合法律？
- 核心流程是否通過 harness 測試？
- 是否踩到紅線？
- 是否符合憲法層？

### 通過條件

- 所有必填項都齊全。
- 核心流程無 crash 通過。
- 沒有違規項目。
- Harness gate 通過。
